## 🎯 核心概念

### 三种时间块状态视觉系统

Trace Timeline 的核心设计是通过**视觉差异**明确区分三种事件状态：

| 状态 | 标识 | 定义 |
|------|------|------|
| **🗓️ 未来计划事件** | PLANNED | 时间在未来的未发生事件（手动添加的时间块、计划的任务） |
| **🔍 未确认 AUTO** | AUTO | 已发生的自动追踪事件（尚未被用户确认） |
| **✓ 已确认** | ✓ | 已发生且已确认的事件（用户手动添加、或已确认的自动追踪） |

> **统一概念验证**:
> ```
> 未来事件 = 计划的事件 = 没有发生的未来事件
>        = 在 tasks 中添加的时间在未来的未发生事件
>        = 在时间线中添加的未来时间的事件
> ```

## 🧩 重叠事件多列布局算法

**问题**: 同一时间可能发生多个事件（如：一边开会议一边写代码），它们需要并排显示，不能重叠。

**算法**: 贪心列分配 (Greedy Column Placement)

### 算法原理

```
1. 按开始时间排序所有块
2. 对每个块，从左到右尝试放入第一列：
   - 不与该列最后一个块重叠 → 放入
   - 重叠 → 试下一列
   - 所有列都重叠 → 新建一列
3. 计算每个块的宽度 = 100% / 最大重叠数
```

### 可视化示例（时间线是纵向垂直的）

```
   ↓ 时间从上到下流动

  9:00  ───────────────────────────
        │  列 0    │  列 1  │  列 2 │
        │████████  │████████│       │
        │ A (Code) │ B (Meet)       │
  9:15  │          │        │       │
        │          │████████│       │
        │          │        ████████│
  9:30  │          │        │ C (设计)
        └──────────┴────────┴───────┘
         <------- 3 列并排 ------->
         每块宽度 = 100% / 3 ≈ 33%
```

**规则就是这么简单：同一时间有 n 个重叠事件 → 每个宽度 = 100% / n**

### 核心代码实现

```typescript
// 检测两个块是否重叠
const blocksOverlap = (a: TimeBlock, b: TimeBlock): boolean => {
  const aStart = new Date(a.startTime).getTime()
  const aEnd = new Date(a.endTime).getTime()
  const bStart = new Date(b.startTime).getTime()
  const bEnd = new Date(b.endTime).getTime()
  return aStart < bEnd && bStart < aEnd  // 半开区间：[start, end)
}

// 贪心列分配
const calculateBlockLayout = (blocks: TimeBlock[]) => {
  const sorted = [...blocks].sort(byStartTime)
  const columns: TimeBlock[][] = []

  // 步骤1：分配列
  for (const block of sorted) {
    let placed = false
    for (let colIndex = 0; colIndex < columns.length; colIndex++) {
      const lastInColumn = columns[colIndex].at(-1)
      if (!blocksOverlap(lastInColumn, block)) {
        columns[colIndex].push(block)
        placed = true
        break
      }
    }
    if (!placed) columns.push([block])
  }

  // 步骤2：计算每个块的宽度和位置
  for (let colIndex = 0; colIndex < columns.length; colIndex++) {
    for (const block of columns[colIndex]) {
      // 找到与这个块重叠的最大数量 = 需要多少列
      const maxOverlap = Math.max(...sorted
        .filter(b => blocksOverlap(b, block))
        .map(b => sorted.filter(other => blocksOverlap(b, other)).length)
      )

      const totalCols = Math.max(maxOverlap, 1)
      const widthPercent = 100 / totalCols

      result.set(block.id, {
        left: colIndex * widthPercent,   // 左偏移
        width: widthPercent - 2,          // 减2px留间隙
        colIndex,
        totalCols
      })
    }
  }
  return result
}
```

### CSS 应用

```tsx
<div
  style={{
    left: `calc(48px + ${layout.left}%)`,
    right: `calc(12px + ${100 - layout.left - layout.width}%)`,
  }}
/>
```

**算法复杂度**: O(n²)，n = 每天事件数（通常 < 20，性能良好）

#### 🏆 今日成就聚合（AI 时间块聚类）

> 解决碎片化工作后「忙了一天不知道自己做了什么」的痛点

**算法逻辑**：
1. 按标题关键词聚类（都包含「客户」「回复」→ 归为一类）
2. 按时间段邻近合并（间隔 < 30 分钟的同分类 → 合并）
3. 生成大事件总结（名称 + 总时长 + 时间范围 + N 个时间块）

const getRecentInterruptedBlock = () => {

### 🏆 今日成就聚合（AI 时间块聚类）

**解决的痛点**：一天都是 5-15 分钟的碎片工作，下班回顾时觉得什么都没做

**核心设计**：**按分类全天聚合**，而非时间邻近聚合。跨时间段的同类活动会被合并成一个大成就。

**真实场景示例**：
| 时间块1 | 时间块2 | 聚合结果 |
|--------|--------|---------|
| 07:00-08:00 健身 | 19:00-20:00 健身 | 🏃 健身 · 2 小时 · 2 个时间块 |
| 09:00-09:10 背单词 | 11:00-12:00 练口语 | 📖 学英语 · 2 小时 10 分 · 3 个时间块 |
| | 19:00-20:00 练听力 | |

**交互设计**：
```
┌─────────────────────────────────────┐
│  🏆  今日成就                          │
├─────────────────────────────────────┤
│  💼  工作                             │
│      8 个时间块 · 总共 4 小时 12 分  │
│      09:00 - 18:38 （跨全天）        │
│  [ ✓ 确认聚合 ]  [ 查看详情 ]         │
├─────────────────────────────────────┤
│  📖  学习英语                        │
│      3 个时间块 · 总共 2 小时 10 分  │
│      09:00 - 20:00 （跨全天）        │
│  [ ✓ 确认聚合 ]  [ 查看详情 ]         │
├─────────────────────────────────────┤
│  🏃  健身                            │
│      2 个时间块 · 总共 2 小时        │
│      07:00 - 20:00 （早 + 晚）       │
│  [ ✓ 确认聚合 ]  [ 查看详情 ]         │
└─────────────────────────────────────┘
```

**AI 聚类算法逻辑**：

```
输入：当天所有时间块（按时间排序）
输出：聚合成的大事件

核心原则：同分类全天聚合，不被时间间隔限制。

步骤：
1. 【第一维：按分类聚合】⭐ 最重要
   - 相同分类的所有时间块 → 归为一类（无视时间间隔）
   - 例：早上健身 + 晚上健身 = 「健身」大类
   - 例：背单词 + 练口语 + 练听力 = 「学习英语」大类
   - 全天所有同分类活动都被合并，不管间隔多久

2. 【第二维：按标题相似度细分】
   - 同分类内，标题包含相同关键词的 → 可进一步细分
   - 例：「回复客户 A」「回复客户 B」「客户电话」→ 「客户问题处理」子分类

3. 【生成大事件总结】
   - 分类名称作为大事件名称
   - 计算总时长 = sum(所有子块时长)
   - 显示时间范围 = 最早开始 ~ 最晚结束（体现跨时段）
   - 显示「N 个时间块」让用户感知到碎片被合并了
   - 点击「查看详情」展开列出所有子时间块
```

**详情展开示例**：
```
📖 学习英语 · 2 小时 10 分 · 3 个时间块
├───────────────────────────────────
│ 09:00 - 09:10  背单词              │
│ 11:00 - 12:00  练口语              │
│ 19:00 - 20:00  练听力              │
└───────────────────────────────────
```

**数据价值**：用户点击「确认聚合」后 → 这个大事件会出现在 Analytics 和日终复盘里，真正让碎片化工作产生「我完成了一件事」的成就感。

> 💡 **设计洞察**：时间邻近不是重点，**类别统一**才是重点。用户不需要知道自己 13:00-13:45 在做什么，他们需要知道「今天我总共工作了多久、学习了多久、健身了多久」。

## 🔗 统一数据模型 & 全局共享架构

> 🎯 **核心架构原则**：**统一命名为「事件 (Event)」。所有条目都是事件，存储在同一个库。不论出现在哪个页面，数据完全一致。**

> |---------|------|------|
> | **时间块** | 同一个 Event 实体 | 有具体开始/结束时间，显示在 Timeline 上 |

>
> **对外只说一个概念：「事件」**。「时间块」「任务」只是用户在不同场景下的习惯叫法，系统内部是同一个东西。

#### 3. 事件 (Event) - 统一数据模型，只有一个实体

> **核心认知**：只有「事件」这一种实体。「任务」「时间块」只是它的两种显示状态，不是两种东西。

```
              ┌───────────────────────────────────┐
              │           事件 (Event)              │
              │    标题、分类、优先级、状态、备注      │
              └───────────────┬───────────────────┘
                              │
              ┌───────────────┴───────────────────┐
              │                                   │
有 scheduledStartTime                    无 scheduledStartTime
（安排了时间）                           （没安排时间）
              ↓                                   ↓
  显示在 Timeline（叫「时间块」）       只显示在 Tasks（叫「任务」）
```

> 💡 最简单的理解：**任务 = 没安排时间的事件，时间块 = 安排了时间的事件**。它们是同一个东西。

```typescript
export interface Task {
  id: string
  title: string
  description?: string
  categoryId?: string       // ⬅️ 与 TimeBlock 共享同一个分类系统
  priority: 'P0' | 'P1' | 'P2' | 'P3'
  status: 'backlog' | 'todo' | 'in-progress' | 'completed' | 'cancelled'
  estimatedMinutes?: number
  actualMinutes?: number
  scheduledStartTime?: string  // ⬅️ 有值 → 在 Timeline 显示为时间块
  scheduledDate?: string       // 没有值 → 在 Tasks 显示为任务
  timeBlockIds?: string[]      // ⬅️ 关联到这个任务的时间块
  userId: string
  createdAt: string
  updatedAt: string
}
```

**统一存储保证**:
```typescript
// ✅ 同一个 ID，处处一致
Task.id = TimeBlock.id = Event.id

// ✅ 同一个标题
Task.title = TimeBlock.title = Event.title

// ✅ 同一个分类
Task.categoryId = TimeBlock.categoryId = Event.categoryId

// ✅ 同一个优先级
Task.priority = TimeBlock.priority = Event.priority

// ✅ 同一个状态
Task.status = TimeBlock.status = Event.status

// 唯一的两个控制字段：
// 1. scheduledStartTime - 控制是否出现在时间线
// 2. status - 控制完成状态
```

> 💡 **核心定义**：**任务 = 时间块 = 事件。它们是同一个东西。**
> - 有 `scheduledStartTime` → 出现在 Timeline（叫「时间块」或「事件」）
> - 没有 `scheduledStartTime` → 只出现在 Tasks（叫「任务」）
> - **有时间的依然是任务**，只是同时显示在两个页面
> - 不需要两张表，只需要一张表，用两个字段控制显示位置

---

#### 显示规则矩阵

> 🎯 **铁则：所有事件都在 Tasks 页面显示，安排了时间的额外在 Timeline 显示。**
>
> - 「安排时间」不是把事件从 Tasks 移到 Timeline
> - 「安排时间」是让事件额外多显示在一个地方
> - Tasks 页面始终是所有事件的完整列表
> - Timeline 只是按时间查看事件的另一种视图

| status 状态 | 有 scheduledStartTime（安排了时间） | 无 scheduledStartTime（没安排时间） |
|------------|----------------------|---------------------|
| `todo` / `in-progress` | ✅ Timeline + ✅ Tasks（待办） | ✅ Tasks（待办） |
| `completed` | ✅ Timeline + ✅ Tasks（已完成） | ✅ Tasks（已完成） |
| `cancelled` | ⚙️ Timeline (可选) + ✅ Tasks（已取消） | ✅ Tasks（已取消） |

> ✅ 总结：**Tasks 显示一切，Timeline 是时间维度的视图补充。** 事件不会因为安排了时间就从 Tasks 消失。

   │  ② isConfirmed = false → 半透明 AUTO 虚线框        │
   │  ③ 用户点击「确认」→ source 变为 'confirmed'       │
   └─────────────────────────────────────────────────┘
```

#### 关键数据流 2: 专注模式 → 时间块写入

```
用户点击「开始专注」（任意页面入口）
  ↓
useFocusStore.isActive = true
useFocusStore.startTime = now
useFocusStore.taskId = selectedTaskId  (如果有关联任务)
  ↓
全局计时器开始，每秒 +1
  ↓
用户点击「结束专注」
  ↓
① 创建 TimeBlock:
   - startTime = focus.startTime
   - endTime = now
   - categoryId = task.categoryId (有关联任务时)
   - source = 'manual'
   - taskId = relatedTask.id
② 更新 Task.actualMinutes += duration
③ 写入 DB，触发 Timeline 自动刷新
```

#### 关键数据流 3: 任务调度 → Timeline 渲染

```
Tasks 页面: 用户拖拽任务到「计划时间」
  ↓
Task.scheduledStartTime = ISO 时间字符串
Task.scheduledDate = YYYY-MM-DD
  ↓
写入 DB
  ↓
Timeline 页面的 useTimelineStore:
  select * from tasks
  where scheduledDate = currentDay
    and status != 'completed'
  ↓
渲染为 🗓️ PLANNED 状态块（斜线纹理 + 虚线边框）
  ↓
到达计划时间 → 自动提醒 "你计划的任务 XX 应该开始了"
```

### 🚨 状态同步边界问题 (必须处理)

#### 问题 1: 分类删除后的脏数据

```
场景: 用户在 Settings 删除了一个 Category
  ↓
Timeline 中所有使用这个分类的 TimeBlock → categoryId 失效
  ↓
结果: 找不到颜色，渲染为白色或崩溃

✅ 解决方案:
  删除分类时的级联策略:
  1. [x] 软删除，不是真删除（推荐 MVP 方案）
  2. [ ] 级联更新所有引用这个分类的 TimeBlock → 改为"其他"
  3. [ ] 级联更新所有引用这个分类的 Task → 改为 null
```

#### 问题 2: 跨页面同时编辑冲突

```
场景: A 页面编辑 TimeBlock.title，B 页面也在编辑同一个块
  ↓
后保存的覆盖先保存的

✅ 解决方案:
  1. 基于时间戳的乐观锁
  2. 或者 MVP 阶段接受最后写入 wins（个人使用场景，冲突概率极低）
```

#### 问题 3: 日期分界点跨天问题

```
场景: 用户凌晨 2 点还在工作
  ↓
如果按自然日 (00:00) 分界 → 这段时间被算到"昨天"
  ↓
用户观感: "我今天刚开始工作，怎么就已经工作 2 小时了？"

✅ 解决方案:
  global.dayCutoffTime = "04:00" （凌晨 4 点）
  计算"今天"范围 = [今天 04:00, 明天 04:00)
  所有页面统一使用这个函数计算日期归属
```

### 🔑 核心设计原则 (全局遵守)

1. **单一真理源**: 同一份数据绝对不存储在两个地方
   - Category 只在 CategoryStore，其他地方只读引用
   - TimeBlock 只在 TimelineStore，Analytics 只读不写

2. **单向数据流**: 永远只允许从上游修改，下游只能读不能写
   ```
   Settings 修改 Category
       ↓
   Timeline 读取新颜色 re-render
   Tasks 读取新选项 re-render
   Analytics 重新计算统计
   ✅ Timeline 永远不修改 Category 定义
   ```

3. **全局样式一致性**:
   - 同一种状态在所有页面视觉表现完全一致
   - AUTO 状态 = 半透明 + 虚线，在 Timeline / Dashboard / Analytics 都一样
   - 用户的认知成本 = 0

┌──────────────────┬─────────────────────────┬─────────────────────────┐
│   左栏: 列表     │   中栏: 详情编辑        │   右栏: 原始参考        │
│                  │                         │                         │
│  [ Block 1 ] ✅  │  ⏰ 9:00 AM - 9:20 AM   │  📊 Overview / 📝 Log   │
│  [ Block 2 ] ⏳  │  📝 标题 [可编辑]       │                         │
│  [ Block 3 ] ⏳  │  📄 AI 生成描述          │  原始活动流水           │
│  ...            │  🏷️ 分类标签             │  (用于对比参考)         │
│                  │  [Accept]  [Reject]     │                         │
└──────────────────┴─────────────────────────┴─────────────────────────┘
```

---

### 左栏：时间块选择器

#### 两种查看模式

##### 1. Blocks 模式（列表视图）
- 每个待确认条目显示：标题 + 时间范围 + Pending 标签
- 支持 `Select all` 批量操作
- 不同状态用颜色区分（Pending = 淡紫背景，Approved = 绿色标记）

##### 2. Timeline 模式（时间轴视图）
- 迷你时间线，可视化查看块的位置和时长
- 支持拖拽调整时间
- 点击块跳转到详情编辑

---

### 中栏：详情编辑（核心）

#### 🔴 重点参考：数据结构扩展

**当前我们只有 `title` 一个字段**，Rize 的设计是**双字段结构**：

| 字段 | 用途 | 内容来源 |
|------|------|----------|
| **Title** | 一句话摘要 | 自动提取 / 用户编辑 |
| **Description** | 详细描述 | **AI 自动生成** / 用户编辑 |

```typescript
// 当前我们的数据结构
interface TimeBlock {
  title: string;        // ← 只有这个
  category: string;
  startTime: string;
  // ...
}

// 建议未来扩展为
interface TimeBlock {
  title: string;        // 短标题（显示在 Timeline 卡片上）
  description: string;  // 长描述（AI 生成的详细说明）
  category: string;
  startTime: string;
  // ...
  
  // 审核状态
  status: 'pending' | 'approved' | 'rejected';
  reviewedAt?: string;  // 审核时间
}
```

#### 编辑区域内容

1. **时间范围** - `9:00 AM - 9:20 AM` + 时长 `20 min`
2. **标题输入框** - 自动填充，用户可修改
3. **描述文本框** - AI 根据窗口标题/URL 生成自然语言描述
   > 示例：`Reviewed a major pull request for a complete UI/UX overhaul and web demo mode in the Merize repository while consulting the product's product design documentation.`
4. **分类标签按钮** - Task / Project / Client 等快速打标
5. **操作按钮**
   - `Accept ⌘+Enter` - 确认（紫色主按钮）
   - `Reject ⌘+X` - 拒绝（灰色次按钮）

### 右栏：原始参考

和左侧列表/Timeline 联动，选中某个块后，右侧显示该时间段内的原始活动流水：
- Overview 聚合视图
- Event Log 精确流水

**作用**: 用户审核时可以对照原始记录，判断 AI 生成的标题/描述是否准确。
