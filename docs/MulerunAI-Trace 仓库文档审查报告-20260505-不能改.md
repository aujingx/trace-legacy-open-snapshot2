## Trace 仓库文档审查报告

### 一、文档整体现状

当前活跃文档 19 个，归档在 `docs/_deprecated/` 下的约 40+ 个。文档经历了至少 3 次大整理（commit `3ddebc7`、`01cd81b`、`1e0920a`），结构从混乱逐步规范化。

### 二、被误删/降级到 deprecated 但仍有高价值的功能设计

以下功能在旧版本中有完整且精心设计的文档，但在当前活跃文档中**要么完全缺失，要么仅被简略提及**：

#### 1. TIMELINE 页面完整设计（2546 行 → 当前无独立文档）

旧版 `TIMELINE.md` 是最详尽的单页设计文档，包含：
- **三状态视觉系统**（PLANNED/AUTO/CONFIRMED）完整 CSS 规范
- **重叠事件多列布局贪心算法**，含完整 TypeScript 实现
- **批量操作栏**（全选当天/全选待确认/批量确认/批量删除）
- **二次确认弹窗规范**
- **上下文智能卡片**（三状态自动切换 -- 被打断恢复/专注正反馈/空闲推荐）
- **今日成就聚合**（AI 时间块聚类，解决碎片化工作的痛点）
- **时间块拆分/合并**（右键菜单操作）

当前状态：Timeline 在活跃文档中没有独立的 feature doc，只在 `IMPLEMENTATION_STATUS.md` 中被简单标记为"框架完成"。

#### 2. AI 智能事件推荐系统（327 行 → 当前无独立文档）

旧版 `TASK_RECOMMENDATION.md` 设计了极具竞争力的推荐引擎：
- **7 维度评分算法**（优先级 35% + 紧迫性 25% + 时长适配 15% + 用户习惯 10% + 进度 8% + 上下文匹配 5% + 情感标签 2%）
- **3 种预设推荐模式**（AI 自动/均衡/优先级优先）
- **用户自定义权重系统**，带滑块 UI
- **机器学习用户习惯**（自动学习时间段+星期+分类偏好）
- **推荐理由解释标签**（"优先级高 · 今天截止 · 已开始"）

当前状态：在 `TASK_MANAGEMENT_DESIGN_FINAL_v4.0.md` 中仅提到"AI 推荐理由标签展示"作为待开发项，推荐算法的完整设计被丢失。

#### 3. 全局智能分心检测系统（229 行 → 当前无独立文档）

旧版 `FOCUS_DETECTION.md` 设计了四层分心检测架构：
- **第一层**：键盘鼠标空闲 + 前台应用切换
- **第二层**：活动分类判断（分心拦截列表匹配）
- **第三层**：用户历史反馈（学习用户标记）
- **第四层**：连续专注正反馈（30 分钟休息提醒）
- **分心弹窗交互**（工作/休息/分心三选一 + 自定义规则）
- **连续专注时长统计**，跨应用不中断
- **分心数据记录与分析**（打断次数、分心应用排行）

当前状态：完全从活跃文档中消失。

#### 4. Execution Guardian 守护者系统（183 行 → 当前仅 Dashboard 简略提及）

旧版 `GUARDIAN.md` 定义了产品的"灵魂"：
- **Now Engine**：确定性规则推荐"现在该做什么"（不依赖 LLM）
- **Morning Ritual**：每日早间规划仪式（设定一天意图）
- **Daily Review**：每晚复盘 + 锚定明天最重要的任务
- **Launch Boost**：专注启动前的摩擦消除
- **核心理念**："不是监督者，是守护者"——教练/助手/朋友的定位

当前状态：Dashboard 文档提到了 Guardian 弹窗的存在，但 Guardian 作为产品灵魂的完整哲学阐述和独立架构文档被移除了。

### 三、我认为最有市场竞争力、值得恢复/强化的功能点

以下按竞争力排序：

#### S 级（核心差异化，直接影响留存率）

| # | 功能 | 来源 | 理由 |
|---|------|------|------|
| 1 | **单任务模式 + AI 推荐** | v3.0 + v4.0 | "敢不敢默认只显示一个任务"——这是 Trace 对抗 Todoist/TickTick 的杀手锏。v4.0 保留了概念但推荐算法的 7 维评分体系没有进入活跃文档，需要恢复。核心洞察"用户打开任务管理不是为了看列表，是为了减少焦虑"非常准确 |
| 2 | **Guardian 日间闭环**（Morning Ritual → Now Engine → Daily Review） | GUARDIAN.md | 早间设定意图 → 全天 Now Engine 引导 → 晚间复盘锚定明天。这个闭环在市面上没有竞品做得好，是 Trace 独有的"温柔陪伴"体验，且不依赖 LLM，100% 本地运行 |
| 3 | **能量等级匹配推荐** | v3.0 + v4.0 | "早上精力旺盛推荐高能量任务，下午低谷推荐低能量任务"——结合学习到的用户习惯曲线，这是真正的 AI 价值而非噱头。v4.0 的能量热力图设计也很好 |

#### A 级（强竞争力，用户感知明显）

| # | 功能 | 来源 | 理由 |
|---|------|------|------|
| 4 | **上下文智能卡片（三状态切换）** | TIMELINE.md | 被打断恢复（"5分钟前你在做XX"）→ 专注正反馈（"已专注47分钟"）→ 空闲推荐。"永远不空白，永远给用户当下最需要的信息"——这个设计理念极好 |

### 四、具体建议

1. **立即恢复**：将 `TASK_RECOMMENDATION.md`（推荐算法）和 `GUARDIAN.md`（守护者体系）从 `_deprecated` 移回活跃文档区。它们的设计质量很高且是产品核心差异化。

2. **合并更新**：`TIMELINE.md` 中的上下文智能卡片、今日成就聚合、三状态视觉系统应该作为正式 feature spec 恢复，而非被当作"旧设计"归档。

3. **补充缺失**：当前活跃文档缺少 Timeline 和 Focus Detection 的独立 feature doc。v4.0 任务管理文档虽然保留了单任务模式和能量等级的概念，但丢失了推荐算法的完整评分逻辑和用户习惯学习机制。

4. **保持归档**：宠物系统、习惯系统、团队模块等功能的旧文档可以继续留在 deprecated，它们不是核心竞争力。

总结：Trace 最大的差异化不是"又一个时间追踪工具"，而是**"理解你不想开始工作的那一刻，温柔地帮你跨过去"**。Guardian 闭环 + 单任务模式 + 能量匹配推荐 + 共情式分心检测——这四个联动起来就是一个完整的"反拖延陪伴系统"，这在市面上没有竞品做到。建议以此为核心重新组织文档和开发优先级。
