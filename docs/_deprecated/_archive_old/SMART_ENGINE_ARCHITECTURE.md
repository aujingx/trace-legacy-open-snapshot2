# Trace 智能系统整体架构
## 「隐形 Agent」统一设计规范

> 问题：Rule Engine、Now Engine、灵感面板、任务推荐、AI 数据分析，它们是什么关系？要不要合并？
>
> 答案：**它们是同一智能系统的不同层次，各司其职，需要统一的事件总线连接，但不应合并成一个大模块。**

## 二、各模块职责对比表

| 模块 | 定位 | 触发时机 | 输出类型 | 时间尺度 | 核心特点 |
|------|------|----------|----------|----------|----------|
| **Rule Engine** | 行为教练 | **实时**（事件发生时） | 小建议、提醒、纠正 | 秒 → 分钟 | 反应式，基于简单规则 |
| **Now Engine** | 任务调度 | **用户主动询问** / 定时 | 推荐下一个任务 | 分钟 → 小时 | 主动式，基于优先级和上下文 |
| **Analytics Engine** | 数据顾问 | **离线**（天/周/月） | 总结报告、洞察 | 小时 → 月 | 归纳式，基于历史数据 |
| **灵感面板** | 统一入口 | **总是存在** | 所有建议的集合展示 | 任意 | UI 容器，不产生内容 |

## 三、详细拆解每个模块

### 3.1 规则引擎 (Rule Engine) - 已完成 v1.0

**定位**：你的私人行为教练，在恰当的时机给你轻轻推一下。

**核心设计**：事件驱动 → 规则匹配 → 产生建议

**已实现的规则示例**：
- 连续专注 2 小时 → 提醒休息
- 频繁切换应用 → 建议开启专注模式
- 创建新任务 → 给出预估时间建议
- 晚上 20:00 → 提醒每日复盘

**特点**：
- ✅ 低延迟，事件发生立即响应
- ✅ 轻量，不需要 AI 模型
- ✅ 可解释性强
- ❌ 不能做复杂推理和长期规划

**输出类型**：`Suggestion` 对象（标题 + 描述 + 置信度）

---

### 3.2 Now 引擎 (Now Engine) - 已存在，需要重构

**定位**：你的私人助理，帮你决定「现在该做什么」。

**这是当前需要明确的：Now Engine 不是规则引擎的一部分！**

Now Engine 回答的问题：
> 「我现在有 30 分钟空窗期，我该做哪个任务？」
>
> 「这个任务被推迟了 3 次了，我是不是该优先处理？」
>
> 「离下班还有 2 小时，我还能完成多少事情？」

**输入数据来源**：
- 任务列表（优先级、预估时间、截止日期）
- 今日已用时间分布
- 用户的精力曲线（历史数据学习）
- 当前时间（是否快下班了、是否饭前）

**输出类型**：推荐的任务列表 + 时间块规划建议

**与 Rule Engine 的区别**：
| 维度 | Rule Engine | Now Engine |
|------|-------------|------------|
| 触发 | 事件自动触发 | 用户点击「现在做什么」/ 定时推荐 |
| 输出 | 1 个具体建议/提醒 | 1-3 个可选任务 + 时间规划 |
| 响应 | 必须毫秒级 | 可以等几秒，用户有预期 |
| 复杂度 | 简单规则 | 可以用排序算法、甚至 AI 模型 |

---

### 3.3 AI 分析引擎 (Analytics Engine) - 待实现

**定位**：你的数据分析师，定期帮你总结和发现问题。

**回答的问题**：
> 「你这周平均每天 10:00-12:00 效率最高，建议把重要工作安排在这个时间段。」
>
> 「你这周刷短视频的时间比上周多了 30%，是有什么原因吗？」
>
> 「你有 5 个任务都预估严重不准，建议看下是哪里出了问题。」

**触发时机**：
- 每日：晚上 21:00 生成每日简报
- 每周：周日晚上生成周报
- 用户主动点击「生成分析」

**特点**：
- 离线计算，不需要实时
- 可以跑较重的算法/调用大模型
- 输出偏向归纳、洞察、趋势

---

### 3.4 灵感面板 - 待实现

**定位**：所有智能建议的统一展示入口。

**它只是一个 UI 容器，本身不产生任何内容！**

灵感面板聚合展示：
- 🎯 **来自 Rule Engine**：休息提醒、分心提醒、预估建议
- ⏰ **来自 Now Engine**：下一个任务推荐、时间块建议
- 📊 **来自 Analytics Engine**：本周效率洞察、趋势报告

**设计原则**：
- 按时间倒序展示，最新的在最上面
- 不同类型的建议有不同的视觉样式
- 用户可以对每个建议进行操作：接受 / 忽略 / 标记为有用/没用
- 所有操作反馈回对应的引擎，用于优化

**与通知的区别**：
- **通知 (Toast)**：打扰性强，只展示最重要的 1-2 条建议，几秒后消失
- **灵感面板**：非打扰性，所有建议都沉淀在这里，用户可以随时查看

## 四、事件分发机制详解

### 4.1 设计原则

**所有模块通过统一事件总线通信，不直接互相调用！**

好处：
1. ✅ **松耦合**：修改 A 模块不需要改 B 模块
2. ✅ **可观测**：所有事件都可以被日志记录和调试
3. ✅ **可扩展**：新增一个模块只需要订阅它关心的事件
4. ✅ **可测试**：可以手动注入事件进行测试

### 4.2 事件类型完整列表

```rust
// ============= 时间信号类 =============
/// 每小时整点触发
HourlyTick { hour: u32 }

/// 每日总结信号（默认 20:00）
DailySummaryTick

/// 每周总结信号（周日 21:00）
WeeklySummaryTick


// ============= 专注会话类 =============
/// 开始专注
FocusSessionStarted { duration_seconds: u32 }

/// 结束专注
FocusSessionEnded { duration_seconds: u32, completed: bool }

/// 专注被打断
FocusSessionInterrupted { reason: String }


// ============= 任务操作类 =============
/// 用户添加了新任务
TaskAdded { task_id: String, title: String, estimated_minutes: u32 }

/// 任务被标记完成
TaskCompleted { task_id: String, actual_minutes: f64 }

/// 任务被修改
TaskUpdated { task_id: String, changed_fields: Vec<String> }

/// 任务被推迟
TaskSnoozed { task_id: String, times: u32 }


// ============= 用户行为类 =============
/// 用户离开电脑（AFK）
UserIdleDetected { idle_seconds: u32 }

/// 用户回到电脑
UserReturned { away_seconds: u32 }

/// 检测到分心模式（频繁切换应用）
DistractionDetected { app_switches: u32, duration_seconds: u32 }

/// 用户打开了 Trace 应用
AppLaunched


// ============= 反馈类 =============
/// 用户接受了建议
SuggestionAccepted { suggestion_id: String, rule_id: Option<String> }

/// 用户拒绝了建议
SuggestionRejected { suggestion_id: String, rule_id: Option<String> }

/// 用户忽略了建议（展示后 5 分钟无操作）
SuggestionIgnored { suggestion_id: String, rule_id: Option<String> }
```

### 4.3 事件流示例

```
用户开始专注
    ↓
事件: FocusSessionStarted
    ├─→ Rule Engine: 重置今日专注时间计数
    └─→ Now Engine: 移除这个任务的推荐
    └─→ UI: 更新专注状态

... 2.5 小时后 ...

用户结束专注
    ↓
事件: FocusSessionEnded (2.5h)
    ├─→ Rule Engine: 检测到长时间专注 → 产生「休息建议」
    │       ↓
    │       事件: NewSuggestionCreated
    │            ├─→ UI Toast: 弹出通知
    │            └─→ UI 灵感面板: 追加一条建议
    │
    └─→ Analytics Engine: 记录今日累计专注时间
    └─→ Now Engine: 重新计算下一个推荐任务
```

## 五、整合方案建议

### 5.1 推荐方案：分模块，统一总线

**不要合并成一个大模块！** 原因：
1. 职责边界清晰，每个模块只做一件事
2. 可以独立迭代、独立测试
3. 性能特性不同（实时 vs 离线）
4. 团队可以并行开发

**整合步骤**：

1. **第一步**：实现统一事件总线（`event_bus.rs`）
   - 发布 / 订阅机制
   - 事件类型统一定义
   - 历史事件日志

2. **第二步**：把现有的 Now Engine 接入事件总线
   - Now Engine 不再直接调用数据库
   - 改为订阅它关心的事件

3. **第三步**：实现灵感面板 UI
   - 聚合展示所有来源的建议
   - 统一的反馈交互

4. **第四步**：实现 Analytics Engine
   - 订阅每日/每周信号
   - 离线计算分析报告
   - 结果存入灵感面板

### 5.2 反馈闭环是关键

所有三个引擎都需要用户反馈来优化：

```
引擎产生建议 → 用户反馈（接受/拒绝）
    ↑                         ↓
    └─ 调整权重 / 置信度  ←──┘
```

但反馈是**每个引擎自己处理**的，因为优化逻辑不同：
- Rule Engine：调整规则的置信度和触发阈值
- Now Engine：调整推荐算法的权重参数
- Analytics Engine：调整洞察的筛选条件
