### 1.3 明确排除的功能（Beta 阶段永不实现）

- ❌ **虚拟宠物系统** - 与核心时间追踪价值无关
- ❌ **习惯追踪** - 超出 Beta MVP 范围
- ❌ **系统托盘 StatusBar** - OS 级交互高风险，V2 后考虑
- ❌ **WanderingDetector 自动走神检测** - 技术复杂度高，假阳性 UX 风险大
- ❌ **AI LLM 深度集成** - Beta 只用确定性逻辑，不依赖外部 API
- ❌ **云同步** - Beta 本地优先，数据存在 Tauri SQLite
- ❌ **团队协作功能** - 个人产品优先，团队版 V3 后考虑
- ❌ **高级 Guardian Analytics 标签页** - Analytics 页只做基础统计
- ❌ **知识卡片系统** - V2 后考虑
- ❌ **会议相关功能** - 会议检测、会议分类覆盖、会议提醒通知、会议结束通知，超出个人时间追踪核心场景
- ❌ **分类开关矩阵** - 不做每个分类有 Focus/Work Hours/Idle/Distraction Blocker 4 个开关的复杂配置，只做基础的分类名称和颜色自定义
- ❌ **分心拦截器 (Distraction Blocker)** - 弹窗拦截、网站屏蔽、成瘾拦截模式，V2 后考虑
- ❌ **时间条目建议 (AI Time Entry Suggestions)** - AI 自动生成时间条目标题、项目、客户建议，复杂度高
- ❌ **分类变更历史 (Categorization History)** - 批量重分类操作记录与回溯
- ❌ **Urge Surfing 模式** - 分心拦截的成瘾干预模式
- ❌ **报告与数据导出高级功能** - CSV 导出、API、日历集成
- ❌ **组织管理与多工作区** - Org Members、团队工作区切换

## 6. 完整产品功能清单

### 6.1 Dashboard 详细功能
- 顶部 4 个彩色统计卡片（蓝/绿/紫/灰，分别是专注时长/目标进度/效率分/活动数）
- Now Engine 推荐卡片（居中，核心视觉元素）
- 2 列数据卡片：Active Tracking + Plan vs Actual
- Daily Review 今日总结卡片
- 底部 3 个功能入口卡片：Review Activities / Priority Tasks / Daily Report
- Morning Ritual 自动弹窗（5:00-18:00 首次打开）
- Daily Review 自动弹窗（20:00 后首次打开）

### 6.2 5-Tab 导航架构

| Tab | 页面 | 核心功能 |
|-----|------|----------|
| 🏠 Dashboard | 首页概览 | 见 6.1 |
| ⏱️ Timeline | 时间轴 | 24h 垂直时间线 + 时间块拖拽 + 右侧详情面板 |
| ✅ Tasks | 任务管理 | 列表/看板双视图 + firstStep + emotionalTag |
| 📊 Analytics | 数据分析 | 基础统计 + 分类饼图 + 周/月趋势 |
| ⚙️ Settings | 设置 | 外观 / 目标 / Guardian / 账户 / 数据 / 快捷键 / 关于 |

### 6.3 Execution Guardian 核心层

#### Now Engine（确定性推荐）
```
优先级顺序：
1. 当前时间块关联的任务
2. 昨天 Daily Review 保存的 tomorrowTopTask
3. 最高优先级未完成任务
4. 最老的逾期任务
```

#### Morning Ritual（晨间仪式）
- 运行时间：每日 5:00-18:00 首次打开
- 内容：问候 + 今日时间块预览 + 未完成工作提示 + 确认今日第一个行动
- 只运行一次，完成后当天不再显示

#### Daily Review（每日复盘）
- 运行时间：每日 20:00 后首次打开
- 内容：今日成就展示 + 计划 vs 实际对比 + 输入明天最重要的任务
- 保存 tomorrowTopTask 给第二天的 Now Engine
- 只运行一次，完成后当天不再显示

#### Launch Boost（启动加速）
- 点击 Now Engine 推荐的"开始"按钮后显示
- 显示任务标题 + firstStep
- 提供快速专注时长选择（25m / 50m / 自定义）
- 确认后进入 Focus Mode

### 6.4 Focus Mode（专注模式）
- **不是独立 Tab**，是覆盖层模态框
- 倒计时显示
- 当前任务名称 + firstStep 提示
- 暂停 / 完成 / 放弃按钮
- 暂停时可保存上下文快照
- 完成后自动更新时间块

### 6.5 Timeline 详细功能
- 24 小时垂直时间轴，每小时一个刻度
- 时间块按分类着色（工作=蓝 / 学习=青 / 娱乐=粉 / 会议=绿 / 其他=灰）
- 红色"当前时间"指示线，实时更新
- 拖拽时间块上下边缘调整时长
- 点击时间块 → 右侧滑出详情面板
  - 编辑标题
  - 选择分类
  - 调整时间
  - 删除按钮
  - 关联到任务
- 空状态提示："今天还没有活动记录"

### 6.6 Tasks 详细功能
- 新建任务表单
  - 标题（必填）
  - 优先级（1-5）
  - 预计时长（分钟）
  - 截止日期（可选）
  - **第一步是什么**（可选 but 强烈推荐）
  - **情绪标签** 😊轻松 / 😐一般 / 😰困难
- 列表视图
  - 垂直滚动列表
  - 每个卡片显示：标题、优先级徽章、第一步（蓝色小字）、情绪标签
  - 完成复选框
  - 删除按钮
- 看板视图
  - 双列：待处理 / 已完成
  - 列顶显示计数
  - 拖拽排序（可选 P2）
  - 空状态虚线提示
- 任务完成后 firstStep 自动隐藏，打钩动画

### 6.7 Analytics 详细功能
- 今日专注总时长
- 本周 vs 上周对比
- 分类分布饼图
- 每日趋势柱状图（最近 7 天）
- 导出按钮（JSON / CSV）

### 6.8 Settings 详细功能
左侧导航分组：
- 🎨 外观：主题（亮色/深色/跟随系统）、背景皮肤
- 🎯 目标：每日专注分钟目标、提醒设置
- 🏷️ 分类管理：自定义活动分类，支持添加/修改/删除、设置分类颜色
  - 默认 5 个分类：工作（蓝）、学习（青）、娱乐（粉）、会议（绿）、其他（灰）
  - 支持新增自定义分类，自定义颜色
  - 支持重命名、修改颜色、删除分类（删除时询问已有时间块迁移到哪个分类）
- 🛡️ Execution Guardian：Morning Ritual 开关、Daily Review 开关、Launch Boost 开关
- 🤖 AI：自动分类开关、API Key 配置、模型选择（文心/豆包）、无 Key 时自动回退到规则分类
- 🔑 账户：个人信息、登录设备、退出登录
- 💾 数据：导出、清除数据、同步状态
- ⌨️ 快捷键：全局快捷键配置
- 📄 关于：版本号、更新日志、隐私政策

### 6.9 系统原生功能（全部已实现）

#### 📊 StatusBar / 系统托盘
- 今日专注时长实时显示在菜单栏
- 快速开始/暂停番茄钟按钮
- 快速打开设置页面
- 退出应用选项
- 动态更新菜单状态

#### 😴 空闲自动暂停检测
- 后台监测用户键盘鼠标活动
- 检测到空闲超过阈值时自动暂停追踪

- 可配置空闲时间阈值
- 避免把离开座位的时间算成专注

#### 🍅 番茄钟（后端驱动）
- Rust 后端驱动的准确计时（不受前端关闭影响）
- 工作时间 / 休息时间自动切换
- 系统通知提醒切换
- 历史记录保存
- Dashboard 可直接控制

#### 🚫 DNS 分心屏蔽
- 可配置屏蔽的网站域名列表
- 专注模式自动启用屏蔽
- 恢复正常模式自动解除
- 系统级生效，所有浏览器都管用

## 9. 开发规则与约束

### 9.1 文档优先级（最高到最低）

当文档之间有冲突时，按以下顺序优先级执行：
1. **`docs/IMPLEMENTATION_PLAN.md`**（本文件）— 功能范围、优先级、按钮说明
2. **`docs/DESIGN_SYSTEM.md`** — 视觉规范、间距、排版、边框阴影规则
3. **`docs/rizescreenshots/README_INDEX.md`** — Rize 竞品截图参考（交互&布局，非配色）
4. **`docs/EXECUTION_GUARDIAN_BLUEPRINT.md`** — 产品意图、核心理念、完整路线图
5. **当前代码库现实** — 实际路由、类型定义、可用命令、已实现功能
6. **历史归档文档** — 只作为灵感参考，永远不是实施真相

### 9.2 Beta 核心目标（已达成）

> 用户感受：**"我正在被温柔地引导进入执行状态"**

1. 打开 Trace → 2. 显示一个推荐任务 → 3. 帮助用户开始 → 4. 帮助用户结束一天
2. 证明这不是"又一个任务清单"
3. Execution Guardian 概念在真实使用中能被感知
4. 不需要聊天对话就能引导行动
5. 日常使用没有明显的导航困惑或流程中断

### 9.3 不可违反的硬约束

1. **先写文档，再写代码** — 任何功能先在这个文档里写清楚用户价值、实现步骤、验收标准，再开始写
2. **完成一个，再下一个** — 不要同时开 3 个坑，一个功能完全做完并测试通过了，再开始下一个
3. **每次提交都要能运行** — git commit 前确保 `npm run build` 能正常编译，没有报错
4. **不做文档里没有的功能** — 如果想加新功能，先更新这个文档，确认优先级再做
5. **绝对不要黑色边框** — 设计系统铁律，最暗的边框就是 `#D6D3CD`
6. **5-Tab 架构不变** — Focus Mode 是覆盖层，不是独立 Tab
7. **本地优先** — 不做云端同步，不做 Flask 后端改动，所有数据存在 Tauri SQLite

### 9.4 代码架构规则

1. **IPC 分层** — 所有 Tauri 命令调用必须走 `src/services/ipc/` 对应文件，不直接在页面 invoke
2. **数据服务统一** — 前端页面调用 `dataService.ts`，不直接调用 IPC
3. **全局状态 Zustand** — `useAppStore.ts` 管理所有跨页面状态
4. **Feature Flag** — 实验性功能用开关包裹，默认关闭，不影响稳定功能

### 9.5 开发纪律

#### 范围控制
- 一次只执行一个阶段
- 不要跳到后面的阶段
- 不要因为"代码就在附近"就扩大范围
- 不要顺手实现 V2/V3 的想法

#### 变更大小
- 每次实施的改动要小，便于审查
- 如果一个阶段需要有意义地修改超过 ~10-12 个文件，先停止并报告

#### 后端纪律
- 不要仅仅因为模型看起来不一致就更改 Flask 后端
- 只有在确认 Beta 关键路径有必要时才动 Python 后端
- 发现后端不一致时报告，不要自动扩大范围

#### 命名纪律
- 代码标识符、文件名、注释：英文优先
- 用户可见文案：在可行时通过现有 i18n 模式处理

#### 安全纪律
- Beta 不做破坏性数据迁移
- 优先使用增量 schema 变更
- 重构导航时保留向后兼容的路由
- 除非有充分理由且替换已可用，否则不删除旧文件

### 9.6 硬停止条件

遇到以下情况必须立即停止：
- `npm run build` 失败，且无法在当前阶段内有信心地修复
- 改动需要破坏性数据迁移
- 某个阶段意外引入了 Flask 后端工作
- 系统托盘 / OS 权限相关工作在核心循环被验证前就出现了

### 9.7 设计一致性规则

1. **同一页面最多 2 种边框风格** — 比如 A2 + A1 是可以的，不要同时用 4 种
2. **边框颜色 = 语义** — 蓝色=主操作，紫色=AI，绿色=成功，珊瑚色=删除/危险
3. **默认风格 = A2** — 2px `#D6D3CD` 边框 + 4px offset 灰色阴影，大部分组件用这个
4. **按钮圆角 12px，卡片圆角 24px** — 严格遵守

### 9.8 功能路线图

#### 🔒 永不加入（与核心理念不符）
- ❌ **虚拟宠物系统** — 与核心时间追踪价值无关，游戏化过度
- ❌ **习惯追踪** — 超出时间追踪定位，属于另一个产品范畴
- ❌ **团队协作功能** — 个人产品优先，团队版 V3 以后单独考虑
- ❌ **会议相关功能** — 会议检测、分类覆盖、提醒通知，需要日历集成权限，且会议是用户无法自主控制的时间
- ❌ **分类开关矩阵** — 每个分类 4 个开关共 120 个配置项，配置过载，普通用户根本用不上。只做基础的分类名称+颜色自定义，不做每个分类的行为开关
- ❌ **分心拦截器 (Distraction Blocker)** — 需要 root 权限、系统代理，容易引起用户反感，且 Guardian 的核心是"引导做正确的事"不是"阻止做错事"
- ❌ **AI 时间条目建议** — 冷启动数据不足，LLM 调用成本高，用户来用 Trace 是为了做事不是美化时间条目
- ❌ **分类变更历史与回溯** — 操作审计级功能，普通用户不需要，增加数据模型复杂度

#### 📅 V2 规划中（Beta 验证后开发）
| 功能 | 说明 | 为什么现在不做 |
|------|------|----------------|
| **WanderingDetector 自动走神干预** | 通过监测用户行为模式判断是否走神，主动提醒回到任务 | 技术复杂度高，假阳性容易引起反感，MVP 不需要"监督"用户 |
| **深度 AI LLM 集成** | 用大模型分析用户行为、给出个性化建议、对话式调整计划 | 确定性规则的 Now Engine 对 MVP 已经足够，LLM 增加成本和延迟 |
| **高级 Guardian Analytics 标签页** | 专门的执行分析页面：拖延倾向、效率曲线、情绪标签趋势、干预成功率 | 普通 Analytics 页已能放基础统计，用户首先是"做"，其次才是"分析自己做得怎么样" |
| **知识卡片系统** | 上下文感知智能提示：继续任务时自动显示上次的上下文、笔记、截图 | 复杂度高，MVP 阶段"上下文快照"文本字段已经够用 |
