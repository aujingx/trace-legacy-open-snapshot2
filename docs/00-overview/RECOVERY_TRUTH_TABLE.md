## 使用规则

1. 在 `P0 核心闭环` 全部通过前，不新增功能，只允许做修复、删减、重命名、接线。
2. 任何代码改动，如果改变了功能真实状态，必须先更新这张表。
3. “页面上看得到”不算实现完成，必须满足：
   - 有明确的数据来源
   - 有真实的读写路径
   - 在桌面包中可验证
4. 设计文档、注释、README 里的描述，不自动等于真实实现。

---

## 当前阶段目标

只做一个目标：

**让桌面正式包完成最小可信闭环**

闭环定义：

1. 打开应用后能开始追踪
2. 能持续记录前台窗口/应用
3. 退出再打开后数据还在
4. 至少有一个主页面能稳定看到真实追踪数据
5. 打包验证流程能覆盖上面 4 项

---

## 正式页面真相

当前产品正式页面以这 6 个为准：

1. `Dashboard`
2. `Timeline`
3. `Tasks`
4. `Analytics`
5. `Settings`
6. `Focus`

补充说明：

- `Statistics` 不是当前正式页面。即使代码文件仍在，也不作为当前产品页面真相。
- `Timeline` 不是边角页，也不是长期可降级页面；它属于核心闭环页面，应该承接真实追踪时间和任务关联。
- `Focus` 指独立的专注/番茄钟页面，不等于 Dashboard 上的快捷启动入口。

## P1 可暂缓功能

这些功能可以保留，但在 P0 闭环未稳定前一律冻结，不继续扩展。

| 功能 | 当前真实状态 | 说明 |
|------|--------------|------|
| 任务管理 | ⚠️ 半可用 | 任务主体在 SQLite；但“今日计划”又有一套 JSON `planned_tasks` 逻辑，属于双轨并存 |
| Time Blocks 计划块 | ⚠️ 半可用 | 有独立 SQLite 表，但和自动追踪活动没有打通 |
| Focus Sessions / Pomodoro | ⚠️ 半可用 | 创建与番茄逻辑存在，但前端对更新/删除还没有完整 IPC 实现 |
| Settings | ⚠️ 半可用 | 一部分设置走 SQLite，一部分还保留 JSON 旧路径与 localStorage 状态 |
| Guardian / Morning Ritual / Daily Review | ⚠️ 冻结 | 可以保留能力方向，但当前弹窗触发方式不可信；不得继续把它们当 onboarding 硬弹 |
| AI 推荐 / Now Engine | ⚠️ 冻结 | 依赖底层任务和活动数据，如果底层失真，推荐结果没有可信度 |
| Habits / Pet | ⚠️ 冻结 | 不是当前救火范围；如果核心数据层未稳定，这些模块只会增加维护噪音 |

## 已失真或误导性实现

这些内容最容易让后续开发继续跑偏，必须明确标记。

| 项目 | 现状 | 风险 |
|------|------|------|
| `dataService.ts` 顶部“桌面端全部走 SQLite”的注释 | 与实际不符 | 误导后续修改者以为活动数据已经完成数据库化 |
| “Timeline = 真实时间线” 的认知 | 与实现不符 | 现在它更像计划块视图，不是自动追踪活动视图 |
| “Statistics 是正式页面” 的认知 | 与当前产品定义不符 | 容易让修复任务围着错误页面集合打转 |
| “Morning Ritual / Daily Review 属于 onboarding” 的认知 | 与当前产品定义不符 | 会让首次使用体验出现错误弹窗，直接破坏新用户体验 |
| “测试通过” | 多数指 web 模式通过 | 容易让人把网页假数据结果误判为桌面包通过 |
| 首次启动引导 | 解除首次启动的关键组件未接入 | 新装包可能永远停在“不自动追踪”的状态 |
| `clearAllData()` | 会清空大量本地状态 | 容易把问题和测试数据/首启状态搅在一起，增加排查成本 |

## 近期决策

在开始改代码前，先明确下面 3 个决策，避免边修边摇摆：

1. **P0 阶段活动数据是否继续暂时以 JSON 为唯一真相**
   - 建议：是。
   - 原因：先把桌面包救活，再做 JSON -> SQLite 迁移；一次只收敛一条主线。

2. **Timeline 是否仍可长期降级为“计划时间块视图”**
   - 建议：否。
   - 原因：短期可以临时止血，但长期正确状态必须是 Timeline 接入真实追踪活动并与任务共存。

3. **Onboarding 是否只保留首次教学**
   - 建议：是。
   - 原因：Morning Ritual / Daily Review / 明日安排类内容不属于首次进入教学，应改为通知式触发。

4. **所有验证是否改为以桌面模式为准**
   - 建议：是。
   - 原因：`npm run dev` 只能算 web demo，不算产品真实验证。

## 建议修复顺序

### Phase 1：止血

1. 修首次启动与自动追踪
2. 明确活动数据唯一来源
3. 统一活动数据目录
4. 确保 Dashboard / Analytics 读到同一份真实数据

### Phase 2：把“看起来能用”改成“真的能用”

1. 修 `getDailyStats` 分类统计
2. 修 `getActivitiesRange`
3. 让 Timeline 接入真实追踪活动并与任务同屏共存
4. 修 onboarding 与通知的边界：首次教学 vs 日间提醒
5. 补一条桌面端冒烟测试流程

### Phase 3：再做迁移，不先重写

1. 设计活动 JSON -> SQLite 迁移策略
2. 一次性迁移活动读写
3. 删除旧 JSON 活动路径
4. 最后再恢复更高层功能开发
