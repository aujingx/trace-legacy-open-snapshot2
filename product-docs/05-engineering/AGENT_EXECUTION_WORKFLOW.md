# Trace Agent 协作工作流

> 最后更新：2026-05-05
>
> 目标：降低开发成本，同时避免多 agent 并行把项目越改越乱。

---

## 1. 角色分工

### 1.0 固定角色映射

- `你` = Jing
- `主控 agent` = Codex
- `执行 agent` = Claude Code

如果后续使用其他执行 agent，默认也归入`执行 agent`角色，不能替代 Codex 做方向定义。

### 1.1 Jing

你负责：

- 决定产品方向
- 设定当前阶段优先级
- 在关键 review gate 做最终取舍

你不需要负责：

- 逐行检查代码
- 手动维护技术一致性
- 追踪每个执行 agent 是否越界

### 1.2 Codex（主控 agent）

主控 agent 负责：

- 把需求变成可执行任务
- 决定这轮只做什么
- 产出验收标准
- 指派执行任务
- 回收结果并 review
- 判断是否合格、是否要返工

主控 agent 不负责“大包大揽全部直接写完”，除非任务很小。

### 1.3 Claude Code（执行 agent）

执行 agent 负责：

- 在明确范围内修改代码
- 只完成被分配的子任务
- 提供验证结果
- 不越权改战略、范围和主文档

执行 agent 不负责：

- 自行改产品方向
- 自行扩展范围
- 顺手优化其他模块

---

## 2. 什么时候适合用多 agent

只在下面两种情况适合：

1. 任务能拆成互不重叠的小块
2. 主控 agent 能定义清楚每块的输入和验收标准

不适合的情况：

- 核心架构还没稳定
- 多个任务会改同一批核心文件
- 需求本身还没说清楚
- 需要先决定真相源而不是直接编码

---

## 3. 标准流程

### Step 1：主控 agent 先做策划

主控 agent 必须先输出：

1. 任务目标
2. 本轮范围
3. 不做内容
4. 影响文件
5. 验收标准

没有这 5 项，不进入执行。

### Step 2：拆成小任务

每个执行任务必须满足：

- 单目标
- 小范围
- 有明确文件边界
- 有可验证结果

好的拆分例子：

- 修活动 IPC 日期参数
- 修统计页分类口径
- 更新桌面验证文档

不好的拆分例子：

- 整理整套架构
- 重做追踪系统
- 统一所有页面逻辑

### Step 3：执行 agent 开发

执行时必须遵守：

1. 只改被分配的文件和问题
2. 不顺手改其他文档
3. 不引入第二套默认实现
4. 改完必须给出验证结果

### Step 4：主控 agent review

主控 agent 回收后必须检查：

1. 是否越界
2. 是否和当前真相表冲突
3. 是否真的解决目标问题
4. 是否留下新的技术债
5. 是否需要返工或缩范围

### Step 5：过 gate 再进入下一轮

一轮任务只有三种结果：

- 通过：可以进入下一轮
- 返工：继续修当前问题
- 冻结：问题超出当前阶段，先记录不做

禁止“差不多就继续叠下一层”。

---

## 4. 任务模板

以后给执行 agent 的任务，建议固定用这个格式：

```md
任务目标：
- 修复什么问题

本轮范围：
- 允许修改哪些文件

本轮不做：
- 明确排除哪些内容

验收标准：
- 怎样算完成

验证要求：
- 至少跑哪些命令
```

---

## 5. 主控 agent 的 review 模板

主控 agent 审查执行结果时，固定看下面几项：

### 5.1 目标是否命中

- 这个 patch 是否真的解决了指定问题

### 5.2 范围是否失控

- 是否碰了未授权模块
- 是否顺手改了文档/架构/样式

### 5.3 真相是否更统一

- 是否减少了双轨实现
- 是否让数据源更清晰

### 5.4 验证是否真实

- 是不是只做了 web 验证
- 桌面行为是否真的验证过

### 5.5 残留风险是否被明确写出

- 哪些问题仍未解决
- 下一轮应该修什么

---

## 6. 哪些任务必须由主控 agent 自己做

以下任务不适合直接丢给执行 agent：

- 决定单一事实源
- 定义阶段范围
- 判断功能保留/废弃
- 核心架构迁移方案
- 对多个执行结果做整合取舍

这些任务本质上是决策，不是体力编码。

---

## 7. 哪些任务适合交给执行 agent

适合委派的任务通常具备下面特征：

- 明确 bug 修复
- 明确单页面修改
- 明确 IPC/接口接线
- 明确测试或文档补全
- 明确非核心模块修补

---

## 8. 成本控制原则

多 agent 的目标不是更快写更多代码，而是：

1. 把主控 agent 的时间花在判断上
2. 把执行 agent 的时间花在小范围实现上
3. 尽量减少返工和范围失控

如果一个任务拆不清楚，宁可由主控 agent 自己先做，不要硬派给多个 agent。

---

## 9. Trace 项目当前推荐模式

现阶段推荐模式：

1. 主控 agent 先看真相表和当前阶段目标
2. 主控 agent 写出一轮最多 1-3 个小任务
3. 执行 agent 分头做互不重叠的小块
4. 主控 agent 回收并 review
5. 更新真相表和状态文档
6. 通过 gate 后再开下一轮

不推荐模式：

- 多个 agent 同时大面积改 `main.rs`、`dataService.ts`、`useAppStore.ts`
- 在没有验收标准时直接让 agent “把这个做完”
- 让执行 agent 自己决定产品取舍

---

## 10. 失败信号

出现下面任一情况，说明协作流程已经失控：

1. 同一个需求被不同 agent 做出两套实现
2. 文档连续被重写但没人维护唯一事实源
3. 执行结果越来越多，但验收问题没有减少
4. 每轮都说“基本好了”，但打包后仍不能用
5. 主控 agent 只转发任务，不做 review

一旦出现这些信号，应立即停止派发新任务，先回到真相表和当前 gate。

---

## 11. 一句话流程

> 你定方向，主控 agent 定边界与验收，执行 agent 只做小范围实现，主控 agent 复查后才算完成。
