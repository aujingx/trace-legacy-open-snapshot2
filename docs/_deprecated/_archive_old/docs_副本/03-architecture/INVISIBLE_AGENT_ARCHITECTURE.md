# 「隐形 Agent」架构设计
## Trace 时间管理 - 自我学习引擎

> **核心设计哲学**：
> 用户不需要知道有一个 Agent 在工作。
> 他们只会觉得：「这个软件越来越懂我，越来越好用」

---

## 一、设计原则

### 1.1 隐形，不打扰
- ❌ 没有「AI 助手」聊天框
- ❌ 没有「Agent 设置」面板
- ❌ 没有「正在学习中...」这类提示
- ✅ 所有智能都体现在 **默认行为的优化** 上

### 1.2 渐进式学习
- Day 1: 基础功能，零智能
- Week 1: 开始有简单的个性化建议
- Month 1: 明显感觉到「它懂我」
- Month 3+: 预判用户行为，主动调整

### 1.3 用户永远有控制权
- Agent 只提建议，不强制执行
- 所有自动调整的行为，用户可以随时「撤销/调整/关闭」
- 用户的每一次操作，都是对 Agent 的反馈信号

## 二、Agent 能力矩阵（全部隐形）

### 2.1 感知层 - 默默观察

| 能力 | 用户感知 | 技术实现 |
|------|----------|----------|
| **效率节律检测** | 「建议把难的任务安排在上午」 | 分析历史数据，找出用户效率最高的时间段 |
| **分心模式识别** | 「下午容易分心，建议开启专注模式」 | 识别频繁切换应用的时间段和诱因 |
| **任务耗时预估** | 预估时间自动填充得越来越准 | 基于历史同类任务的实际耗时，动态调整 |
| **习惯养成追踪** | 不需要用户手动打卡 | 自动检测目标行为是否发生 |

### 2.2 决策层 - 悄悄优化

| 能力 | 用户感知 | 技术实现 |
|------|----------|----------|
| **智能提醒时机** | 「提醒总是在我刚好有空的时候出现」 | 检测到用户切换窗口/空闲时才弹提醒 |
| **番茄钟自适应** | 「今天状态好，自动延长了专注时间」 | 根据最近专注质量动态调整番茄时长 |
| **任务自动重排** | 「计划居然自己变合理了」 | 检测到任务完成情况偏离预期时，自动重排优先级 |
| **过滤无效通知** | 「这个软件从来不会乱弹通知」 | 根据用户当下专注程度，决定是否推送消息 |

### 2.3 进化层 - 慢慢变好

| 能力 | 用户感知 | 技术实现 |
|------|----------|----------|
| **推荐权重自调** | 用得越久，建议越准 | 强化学习：用户采纳建议 → 权重增加；忽略建议 → 权重降低 |
| **个性化分类** | 「它怎么知道这个网站是用来摸鱼的？」 | 用户手动修正分类作为标注数据，自动训练分类器 |
| **异常检测** | 「你今天效率比平时低 30%，要不要休息一下？」 | 对比历史基线，识别异常状态 |
| **长期趋势反馈** | 月度报告里的洞察越来越有深度 | 持续积累用户数据，分析长期变化 |

## 四、学习反馈闭环

```
用户行为 → 提取特征 → 策略引擎 → 产生动作
    ↑                                         ↓
    └──────────── 观察结果 ←──────────────────┘
                        ↓
                  更新策略权重
```

### 奖励信号设计（关键！）

| 用户行为 | 信号类型 | 说明 |
|----------|----------|------|
| ✅ **点击「接受建议」** | **强正奖励** | 策略对了，大幅增加权重 |
| ✅ **按建议执行了任务** | **正奖励** | 隐式接受，比显式点击更真实 |
| ➖ **忽略建议** | **弱负奖励** | 可能建议不对，也可能用户忙 |
| ❌ **明确关闭/拒绝** | **强负奖励** | 策略错了，大幅降低权重 |
| ❌ **卸载/7天不打开** | **灾难性惩罚** | 产品策略出大问题了 |

### 策略置信度机制

- 每个策略都有置信度（0-100%）
- 置信度 < 30%：不执行，继续观察
- 置信度 30-60%：小范围、低影响试探
- 置信度 > 60%：正常执行
- 置信度 > 90%：可以执行更大胆的策略

*原则：用户量越小、数据越少，策略越保守*

## 五、版本迭代路线图

### MVP 阶段（v1.0）- 零智能
- ✅ 基础时间追踪
- ✅ 任务管理
- ✅ 番茄专注
- ✅ 基础统计
- *Agent 目标：静默采集数据，不输出任何建议*

### v1.1 - 规则引擎上线
- 简单的、基于硬规则的建议（所有用户都一样）
- 如：「连续专注 2 小时，建议休息一下」
- *Agent 目标：收集用户对基础建议的反馈*

### v1.2 - 个性化开始
- 基于统计的简单个性化
- 如：根据历史数据填充任务预估时间
- *Agent 目标：验证个性化的效果*

### v1.3 - 强化学习上线
- 多臂老虎机算法，A/B 测试不同策略
- 自动调整建议的时机、内容、频率
- *Agent 目标：开始真正的自我学习*

### v1.4+ - 深度进化
- 更复杂的用户画像
- 异常检测
- 长期趋势分析
- 跨用户迁移学习（类似问题的解法可以借鉴）

## 六、技术实现要点

### 6.1 数据存储
- 用户行为事件日志：SQLite 本地存储
- 特征向量：每周计算一次，增量更新
- 策略权重：JSON 格式，版本化管理
- *所有数据本地存储，不上传服务器（隐私第一）*

### 6.2 计算时机
- **轻量计算**：每次打开应用时，毫秒级完成
- **中等计算**：每天凌晨空闲时计算每日特征
- **重度计算**：每周日后台计算周度特征和策略更新

### 6.3 可解释性（内部用，不展示给用户）
- 每个 Agent 决策都要记录「为什么这么建议」
- 方便调试，也方便未来做「洞察报告」
- 格式：`{策略ID, 置信度, 触发条件, 特征值}`

### 6.4 「出厂重置」机制
- 用户随时可以重置 Agent，让它重新学习
- 也可以选择「只保留数据，重置策略」
- 给用户安全感和控制权

## 七、哲学思考

### 为什么要「隐形」？
1. **降低心理门槛**：用户不需要「学习如何使用 AI」，正常用就行
2. **避免期望落差**：如果说有 AI，用户预期会非常高；如果只是默默变好，都是惊喜
3. **聚焦产品本质**：时间管理的核心是「行动」，不是「跟 AI 聊天」
4. **建立长期信任**：用户会慢慢觉得「这个工具是我的一部分」，而不是一个外来的助手

### 用户心智模型
他们不会说「Trace 的 AI 很聪明」，他们会说：
> **「我用 Trace 之后，时间管理真的变好了」**

这才是产品的最高境界。
