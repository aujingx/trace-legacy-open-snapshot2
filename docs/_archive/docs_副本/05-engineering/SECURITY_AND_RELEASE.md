# 安全与发布控制清单

这份清单面向 Trace 上架、公测、持续收费前的最小安全基线。

## 1. 管控边界

- 登录、付费、订阅状态、团队权限、接口访问令牌必须由服务端裁决。
- 桌面端只保存本地必要缓存，不保存支付密钥、AI 商业密钥、短信密钥、微信支付密钥。
- 自动更新必须使用私钥签名，客户端只内置公钥。
- 用户可导出的只应是自己的数据，服务端开放接口时要按用户身份和租户边界做鉴权。

## 2. 本仓库当前落地方式

- `TRACE_UPDATER_PUBKEY` 不写入仓库，发布时通过 CI Secret 注入。
- Release workflow 在缺少 `TRACE_UPDATER_PUBKEY` 时直接失败，避免生成不可校验更新包。
- 微信 OAuth token 已持久化到本地后端数据库，避免只靠日志占位。
- 组织邀请会落库到 `org_invitations`，并记录通知投递审计到 `notification_events`。
- 微信支付回调会校验签名、校验金额、做幂等处理，并更新订阅有效期。

## 3. 防止“用户脱离你继续用”

- 核心能力必须依赖你的服务端：
  - 登录
  - 订阅校验
  - 支付回调确认
  - 团队协作与云同步
  - 外部 API 调用
- 桌面端即使被反编译，也不能获得：
  - 微信支付 API Key
  - 短信服务密钥
  - AI 商业密钥
  - updater 私钥
- 对“离线可用”能力要有意收敛：
  - 可保留本地采集和只读统计
  - 涉及团队、同步、AI、导出高级能力、开放接口时必须回源校验订阅

## 4. 防拆包与反向工程

- 不把任何商业密钥放进前端/Tauri 包。
- 不在客户端内嵌完整业务规则的“唯一真相”，关键授权在后端重算。
- 发布构建使用最小化 source map 策略；公开包不要带调试日志和测试后门。
- 后端接口全部要求 JWT，并对用户、组织、订单、开放接口做资源级校验。

## 5. 持续收费建议

- 订阅状态以服务端 `subscription_orders` + `subscription_history` + `users.expire_at` 为准。
- 客户端只展示订阅结果，不自己决定是否放行付费能力。
- 后续建议补：
  - 支付宝支付通道
  - 订单查询/补单任务
  - Webhook 重试与告警
  - 个人 API Token / OAuth 应用授权
  - 细粒度 rate limit 与审计日志

## 6. 发布演练

本地执行：

```bash
npm run test:release
RUN_TAURI_BUILD=1 npm run test:release
```

CI Release 前必须保证：

- `TRACE_UPDATER_PUBKEY` 已配置
- 生产环境 `.env` 中所有密钥来自 Secret Manager / CI Secret，不来自仓库文件
- 后端真实环境已验证微信登录、微信支付回调、短信验证码

## 7. 上线 V1 后如何继续开发 V2

目标不是“做完 V1 再停服开发 V2”，而是保持一个稳定发布线，同时让下一个版本持续推进。

### 推荐分支与节奏

- `main`：线上稳定版本，只合入已验证、可发布的改动
- `develop`：下一版本的持续开发线（V1.1 / V2）
- `feature/*`：单个功能开发分支
- `fix/*` 或 hotfix：线上问题修复，直接从 `main` 拉出，修完后同时回合并到 `main` 和 `develop`

### 推荐工作方式

1. V1 发布后，给当前稳定版本打 tag，例如 `v1.0.0`
2. 所有新功能先去 `develop`
3. 若线上版本出问题，从 `main` 拉 hotfix，不要在 `develop` 上直接修线上问题
4. 新版本准备发布时，从 `develop` 做一次发布验证，再合回 `main`

### 对 AI coding agents 的提醒

- 不要在“线上修 bug”的同一次任务里顺手做 V2 功能
- 不要直接在 `main` 上进行大功能开发
- 每次上线前都要明确当前目标是：
  - 稳定版修复
  - 还是下一版本开发

## 8. 自动更新策略（Auto Update）

### 当前仓库现状

这个仓库已经具备自动更新的基础能力：

- Tauri updater 插件已启用
- `src-tauri/tauri.conf.json` 已配置更新端点
- GitHub Release workflow 已要求 `TRACE_UPDATER_PUBKEY`

这说明自动更新不是“以后再研究”，而是**可以落地的已选架构**。

### 推荐自动更新路径

1. 用户安装桌面端
2. App 启动时检查更新
3. 若存在新版本，提示用户更新
4. 对关键版本可升级为强提醒，必要时强制升级

### 建议的版本通道

- `stable`：正式用户通道
- `beta`：测试用户通道

不要长期只用一个“latest”概念，否则测试版和正式版容易混在一起。

### 强提醒与强制更新

建议至少支持两种策略：

- **普通更新**：用户可稍后更新
- **最低版本强制更新**：当旧版本低于服务端允许的最低版本时，必须升级后才能继续使用关键能力

### 对产品负责人的提醒

- 自动更新解决的是“用户拿到新版本”问题
- 但**自动更新本身不能防止白嫖**
- 真正防白嫖要靠“服务端授权 + 最低支持版本 + 试用/订阅状态”

## 9. 如何防止用户下载了旧版本就永久白嫖

这是最关键的提醒：

> **不要把核心商业控制放在客户端本地。**

客户端可以被断网、篡改、卡旧版本。只靠本地代码无法真正防止长期白嫖。

### 必须采用的三层控制

#### 第一层：服务端授权状态

服务端必须决定用户当前是否有资格继续使用关键能力：

- 是否仍在 beta 名单内
- beta 是否过期
- 是否在免费试用期
- 是否已经订阅
- 是否需要续费

### 第二层：最低支持版本

服务端维护一个 `minimum_supported_version`。

如果客户端版本低于这个值：

- 禁止继续使用需要联网/授权的能力
- 提示必须升级

这样即使用户一直留在老版本，也不能无限期享受旧策略。

### 第三层：试用 / beta 过期机制

对于测试阶段，建议不要给“永久免费”。

建议做法：

- beta access 有结束时间
- 免费试用有结束时间
- 到期后：
  - 要么强制升级并重新校验资格
  - 要么降级到只读 / 导出 / 基础本地能力
  - 不再继续提供完整 AI / 云端 / 高级功能

### 最重要的一条

如果某能力未来想收费，就不要让它在 V1 里成为“永久离线可无限使用的完整本地能力”。

否则后面很难收回来。

## 10. Beta 阶段的推荐商业控制策略

你现在是测试阶段，建议用下面这套最实用的组合，而不是一开始就做复杂付费墙。

### 推荐方案

#### 方案 A：测试期免费 + 到期切换

- beta 用户前期免费
- 但 beta 资格有到期时间，例如 14 天 / 30 天 / 内测结束日
- App 每次启动或每 24 小时向服务端刷新一次资格
- 到期后提示：
  - 更新到新版本
  - 或进入候补/正式付费阶段

#### 方案 B：旧版本只保留低价值能力

老版本即使没升级，也只允许继续使用低价值能力，例如：

- 本地历史查看
- 基础活动记录
- 基础导出

以下能力必须回源校验：

- AI 能力
- 云同步
- 付费统计
- 团队能力
- 高级复盘 / 报告

#### 方案 C：最低版本门槛 + 资格刷新

这是最稳的组合：

- 客户端定期刷新授权状态
- 服务端维护最低支持版本
- 旧版本超过门槛后必须升级

这样用户不能靠“永远停在某个免费版本”长期获益。

## 11. 给未来自己的提醒

上线后最容易忘的不是“怎么开发”，而是“怎么收口”。

### 上线后必须长期维护的 4 件事

1. **版本节奏**
   - 稳定版与开发版分开

2. **更新能力**
   - 每个正式版本都可被自动升级覆盖

3. **授权控制**
   - 服务端说了算，不让客户端永久自判

4. **旧版本治理**
   - 设置最低支持版本，不让老版本无限存活

### 一句提醒

> 如果一个版本既能离线完整使用、又没有到期、又没有最低版本限制、又没有服务端授权校验，那么它基本上就等于可以被永久白嫖。
