# 任务管理 - 页面设计文档

## 📋 概述
任务管理的核心矛盾是：**创建任务的门槛越高，用户就越不想创建任务**。

而当任务列表积累了太多任务没完成时，用户会产生「我什么都不想做」的麻木感。

> 我们的任务管理不应该追求「GTD 那样的强大和全面」，而应该追求「让用户愿意创建任务，并且愿意开始做任务」。

## 💡 设计优化讨论点

### 1. 创建任务的门槛 - 「快速添加」模式

**当前设计问题**：
创建任务需要填：标题、预估时间、优先级、分类、截止日期...
心理活动：「好麻烦，等下再填吧」→ 然后就忘了。

**可以讨论的优化方向**：

#### 概念：「一行输入」快速创建
```
[ 今天要做什么？ ]
```
用户只要输入标题，按回车，就创建完成。
其他所有字段（预估时间、优先级等）都是可选的，事后再填也可以。

#### 概念：智能自动补全
```
用户输入：「准备演示」

AI 自动建议：
├─ 预估时间：45 分钟
├─ 优先级：高（因为明天要演示）
└─ 分类：工作
```

用户只要点「确认」就完成了。

**需要讨论的问题**：
- 创建任务的流程应该有多简单？只有标题够不够？
- 智能补全会不会让用户觉得「被安排了」？还是会觉得省心？
- 预估时间如果 AI 猜错了，会不会反而让用户反感？

---

### 2. 任务列表的「压迫感」问题

**核心问题**：
当任务列表有 10+ 个待办时，用户看了就头疼，反而一个都不想开始。
这就是你说的「想到要做各种事情，就无法启动」的感觉。

**可以讨论的优化方向**：

#### 概念：「今天只看 3 个」
不是把所有任务都列出来，而是：
```
✨ 今天推荐你先做这 3 个：

1. 🔴 准备产品演示 (45分钟)
2. 🟡 回复用户邮件 (10分钟)
3. 🟢 更新设计文档 (30分钟)

[ 还有 7 个任务放在后面 ]
```

**设计哲学**：不要让用户在 10 个任务中做选择，帮他选好 3 个，减少决策负担。

#### 概念：「从最简单的开始」
一键按钮：
```
[ 🐣 给我一个 10 分钟就能做完的任务 ]
```

系统自动从任务列表里挑一个预估时间最短的，或者最紧急但最简单的。

**需要讨论的问题**：
- 你在面对一长串任务的时候，是什么样的心理状态？
- 「帮用户减少选择」会不会反而让用户觉得「这不是我的计划」？
- 每天默认只显示 3 个任务，会不会让用户觉得「还有好多事没做完」更焦虑了？

---

### 3. 任务完成的正反馈

**当前设计问题**：
点击「完成」→ 任务划掉 → 消失。
没有成就感。

**可以讨论的优化方向**：

#### 概念：「任务完成总结」
完成一个任务后：
```
🎉 完成！

「准备产品演示」
预估 45 分钟，实际花了 52 分钟。

这是你今天完成的第 2 个任务。
继续保持！ 🌟
```

#### 概念：「雪球效应」可视化
```
🏔️ 今天的进度：

[■■■□□□□□□□] 2/10 任务完成

你已经专注了 2 小时 13 分钟
完成的任务会在时间线上自动标记
```

**需要讨论的问题**：
- 任务完成后需不需要额外的反馈？还是「完成」本身就够了？
- 什么样的反馈会让你觉得有成就感，而不是觉得多余？
- 「实际时间 vs 预估时间」的对比会不会让用户觉得自己效率低？

---

### 4. 「拖延标记」功能

有时候用户明知道应该做某个任务，但就是不想开始。
不是任务难，是「开始」这个动作难。

**可以讨论的设计**：
```
用户右键点击任务 → [ 我有点不想做这个 ]

系统弹出：
我理解 😌
不想开始是很正常的。

👇 你可以试试：

1. [ 就做 5 分钟 ] - 开启一个 5 分钟的专注，做完就可以停
2. [ 先做个更简单的 ] - 推荐另一个简单的任务
3. [ 明天再说 ] - 自动推迟到明天
```

**设计哲学**：不要指责用户拖延，和用户站在一起。
帮助用户绕开「不想开始」的心理障碍，而不是强迫他克服。

**需要讨论的问题**：
- 你觉得这种「承认拖延」的设计有用吗？
- 会不会反而让用户觉得「拖延是被允许的」，然后更拖延了？
- 还是说，这种共情反而能降低心理负担，让用户更容易开始？

## 🎨 参考案例

### Todoist
- 优点：快速添加，快捷键流畅
- 缺点：功能太多太复杂，容易变成任务收集器，反而有压力

### Things 3
- 优点：设计极简，任务创建流畅
- 缺点：没有智能，纯手动管理

### Notion Database
- 优点：极度灵活
- 缺点：管理任务的成本比做任务还高
