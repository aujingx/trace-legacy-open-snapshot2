# 🛡️ Execution Guardian 执行守护者板块

> **状态**: ⏳ 待验证
> **核心定位**: Execution Guardian 不是一个功能，而是整个产品的"灵魂"。它是始终存在的、温柔地引导用户进入执行状态的那个"声音"。
> **设计哲学**: 引导做正确的事，而不是阻止做错事。

---

## 📋 Guardian 三大核心机制

### ✅ 1. Now Engine — 现在该做什么？

**核心问题**: 用户打开应用后茫然四顾，"我现在该做什么？"

**推荐优先级算法**（按顺序）:
```
1. 当前时间块关联的任务（如果正在进行中）
2. 昨天 Daily Review 保存的 tomorrowTopTask（明日之星）
3. 最高优先级的未完成任务
4. 最老的逾期任务（截止日期已过）
```

**设计特点**:
- 确定性规则，不依赖 LLM，不需要 API Key
- 可预测的行为，用户知道"下一个"是什么
- "换一个"按钮让用户有控制权，不会觉得被 AI 绑架

---

### ✅ 2. Morning Ritual — 早上规划一天

**触发条件**:
- 时间：每日 5:00 - 18:00
- 首次打开应用
- Guardian 开关已开启
- 当天还没有显示过

**用户体验流程**:
```
打开 Trace
  ↓
☀️ "早上好！让我们规划今天"
  ↓
显示：日期 + 星期
  ↓
显示：今日已有时间块预览（从日历同步）
  ↓
显示：未完成任务预览（最多 3 个）
  ↓
输入框（预填 Now Engine 推荐）：
  "今天第一个要完成的是：___"
  ↓
点击 [✓ 开始今天]
  ↓
记录 lastMorningRitualDate = 今天
当天不再显示
```

**设计意图**:
> 每天第一件事建立意图。很多人打开电脑先刷邮件/微信，半小时过去了还不知道今天要干嘛。Morning Ritual 强制用户先想清楚"今天第一个行动是什么"，把一整天的基调定下来。

---

### ✅ 3. Daily Review — 晚上复盘明天

**触发条件**:
- 时间：每日 20:00 之后
- 首次打开应用
- Guardian 开关已开启
- 当天还没有显示过

**用户体验流程**:
```
晚上打开 Trace
  ↓
🌙 "今天辛苦了！来复盘一下"
  ↓
🎉 "今天完成了：X 个任务，专注了 Y 小时"
  ↓
📊 "计划 vs 实际：完成度 Z%"
  ↓
输入框：
  "明天最重要的一个任务是：___"
  ↓
点击 [✓ 明天加油]
  ↓
保存 tomorrowTopTask 给明天的 Now Engine
记录 lastDailyReviewDate = 今天
当天不再显示
```

**设计意图**:
> **闭环体验**。早上设定意图，晚上复盘总结，形成完整的一天闭环。
>
> **完成感**。让用户离开应用时带着"今天我做了有价值的事"的积极感受，而不是"还有好多事没做完"的焦虑。
>
> **锚定明天**。保存 tomorrowTopTask，让明天早上的 Now Engine 直接推荐这个任务，减少第二天的决策疲劳。

**触发条件**:

## 📝 Guardian 核心理念

### 不是监督者，是守护者
> Guardian 这个名字是刻意选的，它不是：
> - ❌ 老板 — 盯着你有没有在干活
> - ❌ 老师 — 评判你的效率分数
> - ❌ 狱卒 — 屏蔽网站不让你分心
>
> 它是：
> - ✅ 教练 — 在你需要的时候给你建议
> - ✅ 助手 — 帮你记住上下文，减少认知负担
> - ✅ 朋友 — 早上叫你开工，晚上祝贺你完成
>
> **关键区别**: Guardian 永远给你"建议"，从不给你"命令"。所有建议都可以忽略、跳过、关闭。

### 为什么不用 LLM？
> Beta 阶段 Guardian 是 100% 确定性规则，不调用任何外部 API，原因：
> 1. **速度** — 本地计算，毫秒级响应，没有网络延迟
> 2. **可靠** — 不需要 API Key，不需要网络，随时随地可用
> 3. **可预测** — 用户知道 Guardian 会做什么，不会有意外的"AI 行为"
> 4. **隐私** — 所有数据都在本地，不上传到任何服务器
>
> V2 可能会加 LLM 增强版，但那是"可选升级"，核心 Guardian 永远保持确定性、本地、快速。
