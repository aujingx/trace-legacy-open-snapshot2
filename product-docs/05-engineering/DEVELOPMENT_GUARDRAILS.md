# Trace 开发约束规范

> 最后更新：2026-05-05
>
> 目标：防止需求漂移、双轨实现、假完成、文档与代码脱节。
> 本文档不是通用编码风格说明，而是 Trace 项目的实际开发约束。

---

## 1. 先定原则

### 1.1 单一事实源

每类信息只能有一个主文档：

| 信息类型 | 唯一事实源 | 说明 |
|------|------|------|
| 当前救火期真实实现状态 | `docs/00-overview/RECOVERY_TRUTH_TABLE.md` | 只描述现在真的能不能工作 |
| 产品范围与优先级 | `docs/00-overview/IMPLEMENTATION_STATUS.md` | 只描述当前阶段要做什么，不写幻想功能 |
| 功能设计 | `docs/02-features/*.md` | 每个功能只保留一份活跃设计 |
| 工程约束 | `docs/05-engineering/*.md` | 开发流程、验证流程、协作流程 |

规则：

1. 任何 agent 不得新增第二份“总览”“实现状态”“主计划”文档来替代现有事实源。
2. 如果事实变了，优先更新唯一事实源，而不是新建一份“更准确的版本”。
3. `docs/_deprecated/` 只允许归档，不允许作为当前开发依据。

### 1.2 当前阶段优先级

开发优先级永远按下面顺序：

1. 能不能启动
2. 数据是否真实保存
3. 页面是否读取同一份真实数据
4. 打包后是否仍然工作
5. 新功能与体验优化

如果 1-4 任何一项不稳定，不允许继续扩功能。

### 1.3 “完成”的定义

一个功能只有同时满足下面 4 条，才允许标记为完成：

1. 有明确唯一数据来源
2. 已在目标运行环境验证
3. 文档状态已同步
4. 没有留下第二套并行实现作为默认路径

只满足 UI 可见、注释写了、测试在 web 模式通过，都不算完成。

---

## 2. 功能进入开发前的门槛

任何功能开始编码前，必须先有下面 5 项：

1. 用户价值：这个功能解决什么真实问题
2. 范围边界：这次不做什么
3. 数据来源：读什么、写什么、谁是唯一真相
4. 验收标准：如何判断真的完成
5. 风险说明：最容易和现有系统冲突的地方

如果这 5 项写不清楚，只能继续分析，不能直接写代码。

---

## 3. 文档更新规则

### 3.1 允许修改的文档

- 功能状态变化：更新 `RECOVERY_TRUTH_TABLE.md` 或 `IMPLEMENTATION_STATUS.md`
- 某个功能重新设计：更新对应 `docs/02-features/*.md`
- 新增工程规则：更新 `docs/05-engineering/*.md`

### 3.2 不允许的行为

- 为了“更清楚”新建一份新的总览文档替代旧文档
- 在没有废弃旧文档的情况下复制一份“v2”“最新版”“修正版”
- agent 顺手重写用户刚整理过的战略/设计文档
- 在文档里宣称某功能已完成，但代码和验证没有跟上

### 3.3 变更同步要求

以下变化必须同步更新文档：

- 数据源改变
- 页面定位改变
- 功能冻结/废弃
- 测试方式改变
- 发布方式改变

---

## 4. 代码修改边界

### 4.1 单次任务范围

一次任务只允许解决一类问题：

- 核心启动/持久化问题
- 某个单独页面问题
- 某个单独功能问题
- 某条验证链路问题

禁止在同一个任务里同时做：

- 架构迁移
- UI 重设计
- 文档大整理
- 多模块顺手优化

### 4.2 不允许“顺手”

任何 agent 在执行任务时，不得：

- 顺手改 unrelated 文档
- 顺手统一命名整个项目
- 顺手替换另一套存储方案
- 顺手启用暂未完成的模块
- 顺手补大范围重构

如果发现相关问题，只能记录在输出里，不能顺手做。

### 4.3 双轨实现禁令

禁止长期并存以下情况而不标注：

- JSON 与 SQLite 同时作为默认真相
- web demo 路径和桌面真实路径在文档里不区分
- 新旧页面都被当成正式实现
- 注释宣称走 A，代码实际走 B

如果因为迁移需要暂时双轨，必须写清楚：

1. 谁是当前真相
2. 哪条是过渡路径
3. 什么时候删除旧路径

---

## 5. 核心文件保护名单

以下文件属于高风险区域，任何改动都必须缩小范围并明确理由：

| 区域 | 文件 |
|------|------|
| 桌面启动与追踪主链路 | `src-tauri/src/main.rs` |
| 数据表与迁移 | `src-tauri/src/database.rs` |
| 活动 IPC | `src/services/ipc/activityIpc.ts` |
| 数据总入口 | `src/services/dataService.ts` |
| 全局状态 | `src/store/useAppStore.ts` |
| 首屏启动逻辑 | `src/App.tsx` |

规则：

1. 修改这些文件时，提交说明里必须写清楚“改动目的”和“验证方式”。
2. 如果同一次任务涉及两个以上高风险文件，必须先缩小目标，避免大面积联动。

---

## 6. 验证规则

### 6.1 真实验证优先级

验证顺序必须是：

1. `npm run build`
2. `cargo build --manifest-path src-tauri/Cargo.toml`
3. `npm run tauri build` 或 `npm run tauri dev`
4. 关键路径手动冒烟

`npm run dev` 只能验证 web demo，不算桌面真实验收。

### 6.2 每次改动后的最低验证

| 改动类型 | 最低验证要求 |
|------|------|
| 前端显示/状态逻辑 | `npm run build` |
| Rust / IPC / 持久化 | `cargo build` + `npm run build` |
| 桌面追踪/启动/数据路径 | `npm run tauri build` 或 `npm run tauri dev` |
| 发布流程 | 必须走一次完整打包 |

### 6.3 验收记录

每次任务结束，必须明确说明：

1. 改了什么
2. 没改什么
3. 跑了哪些验证
4. 哪些风险还在

---

## 7. Review 规则

代码审核不以“写出来了”为标准，而以“范围是否受控”为标准。

审核时必须看 5 件事：

1. 是否修改了任务之外的内容
2. 是否引入新的双轨实现
3. 是否让文档与实现更一致，而不是更分裂
4. 是否真的在目标环境验证
5. 是否留下了明确的后续删除/迁移点

如果以上任一项不合格，任务就不算通过。

---

## 8. 什么时候必须停手

出现以下情况时，必须暂停继续开发，先复盘：

1. 同一功能出现第二套默认实现
2. 文档与代码出现明显相反描述
3. web 模式通过但桌面模式失败
4. 新增功能依赖未稳定的数据层
5. agent 开始大面积改文档或跨模块乱动

此时先更新真相表，再决定是否继续。

---

## 9. 当前团队工作方式

Trace 当前适合使用以下模式：

1. Jing 负责方向、范围、取舍
2. Codex 负责策划、拆解、review
3. Claude Code 只负责小范围实现
4. 每轮以验收清单而不是“工作量”作为完成标准

这不是为了增加流程，而是为了降低返工成本。

---

## 10. 一句话规则

> 宁可少做一个功能，也不要再多造一套真相。
